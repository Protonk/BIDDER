# M1 — cycle-walking decision rule

Generated by `m1_cycle_walking.py`. Backend selection in
`coupler.py:Bidder.__init__`: speck32 when
`2^32 <= MAX_CYCLE_WALK_RATIO · P` (i.e., `P >= 2^26 = 67,108,864`), feistel otherwise.

Expected calls-per-output for cycle-walking is `2^32 / P`.
`MAX_CYCLE_WALK_RATIO = 64` caps this; above the cap,
feistel takes over.

## Reading

- **Feistel (below threshold) is roughly constant at ~930 ns/call**
  across `P ∈ [2^16, 2^25]`. The work per call doesn't depend on
  `P`; the Feistel rounds are fixed.

- **Speck32 with cycle-walking** starts at ~2820 ns/call at the
  worst-case `P = 2^26` (expected ratio 64) and drops monotonically
  as `P` grows. The cost tracks the expected ratio: at ratio 1
  (`P` near `2^32`), Speck32 lands at ~910 ns/call, comparable to
  Feistel.

- **The implementation's threshold is conservative.** The code
  switches to Speck32 at `P = 2^26` (`MAX_CYCLE_WALK_RATIO = 64`),
  but at that threshold Speck32 with cycle-walking is **~3× slower
  than Feistel**. The actual *throughput-optimal* crossover is
  near expected ratio ≈ 1, i.e., `P ≈ 2^32 / few`. The conservative
  threshold trades a known small region of suboptimal Speck32
  performance for code simplicity (a single fixed cap).

- **Recommendation for §4.3.** The paper should report the table
  honestly: the *correctness* threshold is `P = 2^26` (Speck32
  saturates the cycle-walking budget there), the *throughput-
  optimal* threshold is `P ≈ 2^31`, and the implementation chose
  the correctness threshold. Users who want best throughput across
  all `P` can either tune `MAX_CYCLE_WALK_RATIO` or accept the
  ~3× factor on a narrow band. This is the §4.3 paragraph's key
  numerical finding.

- **What the §4.3 paragraph cites from this table.** The row at
  `P = 2^26` (~2820 ns) and the row at `P = 2^31` (~954 ns) plus
  the constant Feistel floor (~930 ns) tell the story in three
  numbers.
