# §7.3 — Reproducible cross-validation, exact fold sizes

Generated by `replication/use_case_03_cross_validation.py`. Cipher-leveraged use case (§4.4 permutation contract is what's load-bearing; §3 substrate is unused).

Construction: `fold(i) := ⌊BIDDER.cipher.at(i) · k / n⌋`. Since the cipher is a bijection of `[0, n)` (§4.4), the set `{cipher.at(i) : i ∈ [0, n)}` equals `{0, …, n-1}` as a multiset. The fold cardinalities are therefore independent of which records land in which fold — every fold has either `⌊n/k⌋` or `⌈n/k⌉` elements.

## Fold-size deviation panel

Max deviation = max(|fold_size − target|) where target ∈ {⌊n/k⌋, ⌈n/k⌉}. BIDDER is the cipher; hash row is SHA-256-based bucket assignment, `fold(i) := SHA-256(salt || i) mod k`.

| (n, k) | ⌊n/k⌋ | ⌈n/k⌉ | BIDDER max dev | hash max dev (worst of trials) | hash n_trials |
|---|---|---|---|---|---|
| (1,000, 5) | 200 | 200 | **0** | 46 | 1000 |
| (10,000, 7) | 1,428 | 1,429 | **0** | 129 | 1000 |
| (100,000, 10) | 10,000 | 10,000 | **0** | 358 | 1000 |
| (1,000,000, 10) | 100,000 | 100,000 | **0** | 1,113 | 100 |
| (1,000,000, 100) | 10,000 | 10,000 | **0** | 401 | 100 |

*BIDDER row: deviation is `0` when `k | n`, `1` when `k ∤ n` — fold sizes are tight by construction.* Hash row shows the worst deviation across the trial budget; the one-σ band is `sqrt(n · (1/k) · ((k-1)/k))` and the expected max over independent draws of the n / k multinomial scales as `O(sqrt(n log k / k²))`.

## Reading

- **Headline.** BIDDER's fold-size deviation is at most 1, by construction. Hash-based assignment can be off by hundreds at moderate `n` (binomial sampling noise) and grows as `sqrt(n)`. For applications where fold-size balance is load-bearing (balanced CV variance bounds, fairness audits, cross-implementation reproducibility where deviation patterns must not drift between hash functions), BIDDER's exact-by-construction guarantee is the difference.
- **Reproducibility.** Same key + same `(n, k)` → same partition. Different keys → different partitions of the same exact sizes.
- **Streaming.** `BIDDER.cipher.at(i)` is `O(1)` per record (§4.4 stateless contract); no materialised permutation of `[0, n)`. Workable at `n` larger than RAM, where `numpy.random.permutation(n)` is not.
- **Comparator note.** Hash-based bucket assignment is the simpler choice when `O(sqrt(n/k))` fold-size deviation is acceptable; `numpy.random.permutation(n)` with a seed plus contiguous partition matches BIDDER on exactness when the dataset fits in memory (streaming requirement is what BIDDER is for).
