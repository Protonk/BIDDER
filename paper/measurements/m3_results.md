## Caveats on the throughput numbers

- **BIDDER's per-element cost is ctypes overhead, not C work.** The
  Python wrapper makes one ctypes call per `at(i)`. Benchmarking
  BIDDER through the Python wrapper at ~5 µs/element measures
  Python→C call overhead, not the underlying Speck32 / Feistel
  cost (which is sub-microsecond per element when called from C
  directly). Users wanting peak BIDDER throughput call the C
  kernel from C; the Python wrapper is for convenience, not
  throughput.
- **BIDDER's streaming property is masked here.** The test harness
  builds a Python list `[block.at(i) for i in range(P)]`, which is
  O(P) memory. In true streaming use, BIDDER's runtime memory is
  O(1); the comparators that materialise a full permutation
  (`numpy.random.permutation`, `random.shuffle`, `sort-by-iid-key`)
  cannot do this and so cannot scale beyond RAM.
- **The capability matrix is the load-bearing finding**, not the
  raw ns/elem column. BIDDER is the only comparator with all of
  {keyed, reproducible, streaming/random-access, arbitrary `P`,
  zero extra dependencies}.

## Reading

- **BIDDER vs numpy.random.permutation.** numpy is fastest per element at every `P` (mature C inner loop on a contiguous array), but it materialises the full permutation of length `P`. BIDDER is streaming / random-access: no `O(P)` memory required, suitable for `P` larger than RAM.
- **BIDDER vs random.shuffle.** Same in-memory limitation as numpy; pure-Python shuffle is slower than the C BIDDER cipher at large `P`.
- **BIDDER vs sort-by-iid-key.** Sort-by-key needs all `P` keys in memory plus an `O(P log P)` sort — no streaming, higher memory overhead than `numpy.random.permutation`. Acceptable when you wanted argsort-style output anyway.
- **BIDDER vs i.i.d. with replacement.** i.i.d. is fast and streaming but is *not a permutation*; the finite-population correction does not apply, so without-replacement variance reduction is lost. Listed for completeness as the "do nothing" baseline.
- **BIDDER vs FF1 / FF3-1.** Not benchmarked here. *Cards-face-up:* FF1 / FF3-1 with AES are expected to **beat BIDDER on FPC realisation tightness** (M2's gap goes to ~32× at moderate `P`; FF1 with a strong block cipher should land near 1×) and **beat BIDDER on cryptographic PRP strength** (AES-128 vs Speck32-bit, with the latter not designed for cryptographic security at single-block scale). FF1 / FF3-1 will lose to BIDDER on **per-call cost** (AES-128 vs Speck32 / 8-round Feistel) and **dependency footprint** (AES + FPE library vs zero-deps C). The qualitative trade-off — FF1 wins on quality, BIDDER wins on lightness — is committed in the paper. Numerical confirmation is deferred: see `paper/DEBTS.md` D1.

- **BIDDER vs `secrets.SystemRandom().shuffle()` and `os.urandom`-driven sort-by-key.** Real-randomness baselines for applications needing cryptographically unguessable output. **BIDDER doesn't compete on randomness quality** — these are not keyed reproducible bijections, they are CSPRNG-driven shuffles. The paper acknowledges this directly rather than implying BIDDER is a substitute for them. Not benchmarked here; see DEBTS.md D2, D3.
- **What §7.4 cites from this table.** The capability matrix is the single most informative figure; the throughput numbers second. The use case in §7.4 (small-`P` format-preserving permutation where FF1/FF3 are overkill) lands as: BIDDER's streaming + random-access + zero-deps trio is what no other comparator delivers simultaneously.
