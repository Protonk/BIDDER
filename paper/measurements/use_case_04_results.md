## Capability matrix (from M3)

Five axes, six rows. The headline finding is *qualitative*: BIDDER is the only comparator with all of {keyed, reproducible, streaming/random-access, arbitrary `P`, zero deps}.

| comparator | keyed | reproducible | streaming | arbitrary P | extra deps |
|---|---|---|---|---|---|
| BIDDER (cipher) | yes | yes | yes (random-access) | yes (P ≤ 2³² − 1) | none (~300 lines C) |
| numpy.random.permutation | via seed | yes | no (in-memory) | yes | numpy |
| random.shuffle | via seed | yes | no (in-memory) | yes | stdlib only |
| sort-by-iid-key | via seed | yes | no (sort is global) | yes | numpy |
| i.i.d. (replacement) | via seed | yes | yes | yes | numpy or stdlib |
| FF1 / FF3-1 (D1) | yes | yes | yes (random-access) | yes | AES + FPE library |

## Throughput across three workloads (§6.4 taxonomy)

Workloads measure different things; rows tagged so readers compare like to like.

| workload | BIDDER | FF1 (AES-128) | numpy.random.permutation | random.shuffle | sort-by-iid-key |
|---|---|---|---|---|---|
| (1) single random-access call (M4) | ~940 ns/call (cipher) | n/a | n/a | n/a | n/a |
| (2) materialise full perm at P = 1M (M3 + D1) | 5,366 ns/elem | 100,202 ns/elem | 268 ns/elem | 1,889 ns/elem | 407 ns/elem |
| (3) C-direct kernel (D4) | 38 ns/call (Feistel) | not benchmarked C-direct | n/a | n/a | n/a |

*FF1 / BIDDER through the same Python wrapper:* FF1 is ~19× heavier at P = 1M, ~24× at P = 100k, ~29× at P = 10k (cycle-walking ratio higher at smaller P). FF1 lands at FPC ratio ~0.92 at the M2 cells; BIDDER at 6.8–32× the ideal. The trade is **~25× per-call cost for ~20× tighter FPC**.

## Reading

- **Headline.** BIDDER is the only comparator with all of {keyed, reproducible, streaming, arbitrary `P`, zero deps}. The C-direct kernel is sub-µs (38 ns/call Feistel); through the Python wrapper, ~5 µs/elem when materialising the full permutation. FF1 / FF3-1 with AES is the right choice when cryptographic-strength PRP and tight FPC realisation are required and ~20× higher per-call cost is acceptable.
- **Use case scope.** Anonymising customer IDs in `[0, 10⁶)` for a deterministic test corpus; randomising row order in a finite dataset for staged processing where the same key reproduces the order on re-run; deterministically permuting a fixed-size token vocabulary across runs of an experiment. None of these are adversarial settings; cryptographic-strength PRP is not load-bearing.
- **Comparator notes.** `numpy.random.permutation` wins workload (2) when the dataset fits in memory; FF1 wins on FPC tightness and PRP strength at higher per-call cost; `secrets.SystemRandom().shuffle()` is the right tool when CSPRNG-grade unguessability matters and reproducibility doesn't.
