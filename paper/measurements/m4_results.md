## Reading

- **Both paths show roughly P-independent throughput** through the Python wrapper. The per-call cost is dominated by the ctypes round-trip (~1 µs); the underlying C kernel is sub-microsecond per call.
- **Sawtooth and cipher land within ~30% of each other** through the Python wrapper. The cipher path (Speck32 / Feistel + cycle-walking) is ~940 ns/elem; the sawtooth path (`c_K = qn + r + 1`, one divmod) is ~1300 ns/elem. The unexpected ordering — sawtooth slightly slower — is wrapper-specific (different ctypes argument and return-value handling for the two C entry points), not a property of the underlying C kernels. The C-level cost difference is tens of ns vs hundreds of ns; both are swamped by the Python→C round-trip.
- **For high-throughput applications**, drop into the C kernel directly. The C functions `bdo_block_at` and `bdo_nprime_at` have no Python overhead and are limited only by Speck32 / divmod cost.
- **§6.4 cites from this row**: a single number per path (from the §6.4 row block above), with a note on the wrapper-overhead caveat.
